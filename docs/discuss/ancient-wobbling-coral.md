# Mechanical Spec Enforcement for Threadwork

## Context

OpenAI's "Harness Engineering" article reveals that the biggest differentiator in their agent-driven development is **mechanical enforcement** — deterministic rules that fail the build when agents violate architectural boundaries. Currently, Threadwork's specs are advisory (prose injected as context). Agents are *told* to follow specs but nothing *verifies* they did. This plan adds:

1. A **rules engine** in specs for simple checks (grep, imports, naming)
2. A **structural test gate** for complex invariants
3. An **enhanced discuss-phase** that captures enforcement rules + review criteria from users
4. A **spec compliance quality gate** that runs in the Ralph Loop alongside lint/type/test
5. **Binding constraints** generated by plan-checker and verified post-execution

## Part 1: Spec Rules Engine

### 1.1 Extend spec frontmatter with `rules` field

**File**: `lib/spec-engine.js`

Add support for a `rules` array in spec YAML frontmatter. Each rule has:

```yaml
---
domain: backend
specId: SPEC:auth-001
name: JWT Best Practices
tags: [auth, jwt]
rules:
  - type: grep_must_exist
    pattern: "RS256"
    files: "src/lib/auth*.ts"
    message: "JWT must use RS256 algorithm per SPEC:auth-001"

  - type: grep_must_not_exist
    pattern: "require\\(['\"]jsonwebtoken['\"]\\)"
    files: "src/**/*.{ts,js}"
    message: "Use jose library, not jsonwebtoken (SPEC:auth-001)"

  - type: import_boundary
    from: "src/services/**"
    cannot_import: ["src/ui/**", "src/components/**"]
    message: "Service layer cannot import from UI layer (SPEC:arch-001)"

  - type: naming_pattern
    pattern: "^use[A-Z]"
    files: "src/hooks/**/*.ts"
    target: "export_names"
    message: "Hooks must start with 'use' prefix (SPEC:react-001)"

  - type: file_structure
    must_exist: ["src/services/*/index.ts", "src/services/*/types.ts"]
    message: "Services must have index.ts and types.ts (SPEC:arch-002)"
---
```

**Rule types** (start with 5, extensible):

| Type | What it checks | How |
|------|---------------|-----|
| `grep_must_exist` | Pattern MUST appear in matching files | `grep -r` with glob filter |
| `grep_must_not_exist` | Pattern must NOT appear in matching files | `grep -r` inverse |
| `import_boundary` | Files in `from` glob cannot import from `cannot_import` globs | Parse import statements |
| `naming_pattern` | Exported names in matching files must match regex | Parse exports, test regex |
| `file_structure` | Required file patterns must exist | Glob check |

### 1.2 Add `loadRules()` and `evaluateRules()` to spec-engine.js

```javascript
// Load all rules from all active specs (confidence >= 0.7 or in main spec dirs)
loadRules(phase) → Rule[]

// Evaluate rules against current working tree
// Returns: { passed: boolean, violations: Violation[] }
// Each Violation: { specId, rule, message, files: string[], evidence: string }
evaluateRules(rules, projectRoot) → RuleResult
```

**Key files**: `lib/spec-engine.js` (extend), new helper `lib/rule-evaluator.js`

## Part 2: Structural Test Gate

### 2.1 Convention: `.threadwork/structural-tests/` directory

Users (or the discuss-phase) can create structural test files — plain JS/TS files that export a `check()` function:

```javascript
// .threadwork/structural-tests/no-circular-deps.js
export const name = 'No Circular Dependencies';
export const specId = 'SPEC:arch-001'; // optional link to spec

export async function check(projectRoot) {
  // Run madge or custom logic
  const { execSync } = await import('child_process');
  try {
    execSync('npx madge --circular src/', { cwd: projectRoot });
    return { passed: true };
  } catch (e) {
    return {
      passed: false,
      errors: [`Circular dependencies found: ${e.stdout}`]
    };
  }
}
```

### 2.2 Add `runStructuralTests()` gate to quality-gate.js

```javascript
// Discover and run all files in .threadwork/structural-tests/
// Each must export: { name, check(projectRoot) → { passed, errors[] } }
runStructuralTests() → GateResult
```

Register as a new gate in the quality gate runner alongside typecheck/lint/tests/build/security.

**Key files**: `lib/quality-gate.js` (extend)

## Part 3: Spec Compliance Quality Gate

### 3.1 New gate: `runSpecCompliance()`

**File**: `lib/quality-gate.js`

Combines Part 1 (spec rules) and Part 2 (structural tests) into a single gate:

```javascript
async function runSpecCompliance(options = {}) {
  const specEngine = options.specEngine || new SpecEngine();
  const projectRoot = options.projectRoot || process.cwd();

  // 1. Load spec rules
  const rules = specEngine.loadRules();
  const ruleResult = await evaluateRules(rules, projectRoot);

  // 2. Run structural tests
  const structResult = await runStructuralTests(projectRoot);

  // 3. Merge results
  return {
    gate: 'spec-compliance',
    passed: ruleResult.passed && structResult.passed,
    violations: ruleResult.violations,
    structuralFailures: structResult.errors,
    skipped: rules.length === 0 && structResult.tests === 0,
    reason: rules.length === 0 ? 'No spec rules or structural tests defined' : undefined
  };
}
```

### 3.2 Register in `runAll()` and Ralph Loop

Add `spec-compliance` to the gate sequence. Run it **after** lint/type/test (so basic errors are caught first):

```
Gate order: typecheck → lint → tests → spec-compliance → build → security
```

In `quality-config.json`:
```json
{
  "spec-compliance": { "enabled": true, "blocking": true }
}
```

### 3.3 Extend `buildRemediationBlock()` for spec violations

When spec-compliance fails, the remediation block includes:
- The exact rule that was violated (type, pattern, message)
- The spec ID and spec name
- The file(s) where the violation was found
- A concrete fix template based on rule type

Example remediation for `grep_must_not_exist`:
```javascript
{
  primary_violation: "SPEC:auth-001 rule violated: Use jose library, not jsonwebtoken",
  relevant_spec: "SPEC:auth-001 | backend/jwt-best-practices",
  fix_template: "Replace `require('jsonwebtoken')` with `import { SignJWT } from 'jose'` in src/lib/auth.ts",
  learning_signal: "spec-rule:auth-001:grep_must_not_exist:jsonwebtoken"
}
```

**Key files**: `lib/quality-gate.js`, `hooks/subagent-stop.js`

### 4.2 Auto-generate spec rules from answers

After the user answers questions 6-8, the discuss-phase agent automatically:

1. Creates or updates a spec file (e.g., `specs/enforcement/phase-N-rules.md`) with machine-checkable rules derived from the answers
2. Example: User says "services cannot import from UI" → generates:
   ```yaml
   rules:
     - type: import_boundary
       from: "src/services/**"
       cannot_import: ["src/ui/**", "src/components/**"]
       message: "Service layer cannot import from UI layer"
   ```
3. User can review and edit the generated rules before planning begins

### 4.3 Store review criteria in CONTEXT.md

Question 9's answers go into a new `## Review Criteria` section in CONTEXT.md:

```markdown
## Review Criteria
- Error messages must be user-facing (no stack traces in responses)
- All database queries must use parameterized queries
- No hardcoded secrets or API keys
- Functions should not exceed 50 lines
```

This section is injected into the verifier agent, enabling **semantic review** against user-defined criteria.

**Key files**: `templates/commands/tw-discuss-phase.md`, `templates/agents/tw-researcher.md` (if researcher handles discuss)

## Part 5: Binding Constraints (Plan-Checker → Executor)

### 5.1 Plan-checker generates binding constraints

**File**: `templates/agents/tw-plan-checker.md`

Enhance Dimension 6 (Spec Compliance). After validating plans against specs, the plan-checker:

1. Extracts all applicable spec rules for each plan's file targets
2. Generates a `<constraints>` block per plan:

```xml
<plan id="PLAN-1-2">
  ...
  <constraints generated-by="plan-checker" timestamp="...">
    <constraint specId="SPEC:auth-001" rule="grep_must_exist">
      Files matching src/lib/auth*.ts MUST contain pattern "RS256"
    </constraint>
    <constraint specId="SPEC:arch-001" rule="import_boundary">
      Files in src/services/** MUST NOT import from src/ui/**
    </constraint>
  </constraints>
</plan>
```

### 5.2 Executor receives constraints at spawn

**File**: `hooks/pre-tool-use.js`

When injecting context into executor agents, include the `<constraints>` block from the plan XML. The executor sees both:
- The routing map (for on-demand spec fetching)
- The binding constraints (non-negotiable rules)

### 5.3 Verifier checks constraint compliance

**File**: `templates/agents/tw-verifier.md`

Add a new verification dimension: **Constraint Compliance**. For each plan's `<constraints>`, verify the constraint was satisfied by reading the actual code. Report as:

```markdown
## Constraint Compliance

| Constraint | Spec | Status | Evidence |
|-----------|------|--------|----------|
| auth*.ts contains RS256 | SPEC:auth-001 | PASS | src/lib/auth.ts:23 |
| services/ no UI imports | SPEC:arch-001 | PASS | grep found 0 violations |
```
