# Threadwork v0.3.0 — Update Blueprint

> **Version**: 0.3.0
> **Status**: Upgrade spec — applies to existing Threadwork projects
> **Prerequisite**: Threadwork v0.2.x already initialized
> **Upgrade command**: `threadwork update --to v0.3.0`

---

## Why This Update Exists

Five operational gaps emerged from real-world use of Threadwork v0.2.x that are addressable without changing the core execution architecture.

**Gap 1 — Manual .gitignore management**
Developers initializing Threadwork on a new project had to manually decide which `.threadwork/` files to commit. Operational state files (checkpoint, ralph-state, token-log) were regularly committed by accident, polluting git history with noise. Memory files (journals, handoffs, specs) were sometimes excluded, breaking cross-session continuity. This should be automatic.

**Gap 2 — Context window ambiguity**
The 200K vs 1M context model choice was left entirely to the developer with no guidance from the tool. Expensive 1M context was sometimes used by default when 200K was sufficient, or developers were unaware when a task was approaching 200K limits. The tool should surface this clearly.

**Gap 3 — Token count is not the same as cost**
Threadwork's budget system tracked tokens, but a developer mixing Haiku, Sonnet, and Opus agents was flying blind on actual API cost. 100K tokens on Opus costs ~50× more than 100K tokens on Haiku. The budget system needed to track dollars alongside tokens, and the session default needed recalibration for the 200K model tier.

**Gap 4 — No model tier management**
The model profile system (Haiku/Sonnet/Opus per agent) was defined in the agent roster but had no runtime enforcement, no switch notifications, and no audit trail. Developers couldn't see which model ran which task or approve/deny model tier changes.

**Gap 5 — Blueprint drift had no tooling**
Projects often evolve beyond their original blueprint mid-implementation. There was no mechanism to analyze what a blueprint change would cost, categorize the impact, or choose a migration strategy. Developers either rebuilt from scratch or made ad-hoc changes with no cost visibility.

## Change 1: .gitignore Automation at `threadwork init`

### What changes

`threadwork init` now automatically appends a clearly delimited block to the project's `.gitignore`. If `.gitignore` doesn't exist, it is created. The block is idempotent — running `threadwork init` twice does not duplicate entries.

### .gitignore block written by init

```
# Threadwork — operational state (auto-generated, do not commit)
# Generated by threadwork init. Safe to edit below this line.
.threadwork/state/checkpoint.json
.threadwork/state/active-task.json
.threadwork/state/hook-log.json
.threadwork/state/ralph-state.json
.threadwork/state/token-log.json
.threadwork/state/phases/**/execution-log.json
.threadwork/worktrees/
.threadwork/backup/
# End Threadwork operational state
```

### What IS committed (memory + intent)

The following are explicitly NOT in the gitignore and should be committed:

- `.threadwork/state/project.json` — project config and metadata
- `.threadwork/state/quality-config.json` — quality gate configuration
- `.threadwork/state/phases/**/CONTEXT.md` — phase discussion outputs
- `.threadwork/state/phases/**/plans/*.xml` — execution plans (with decisions)
- `.threadwork/state/phases/**/VERIFICATION.md` — verification results
- `.threadwork/state/phases/**/UAT.md` — UAT checklists
- `.threadwork/state/phases/**/token-summary.json` — phase token+cost variance (NEW, see Change 3)
- `.threadwork/specs/` — full spec library
- `.threadwork/workspace/journals/` — session journals
- `.threadwork/workspace/handoffs/` — session handoffs
- `.threadwork/store/` — cross-session Store (project-level cache)
- `THREADWORK.md` — framework guide at project root

## Change 2: 200K Context Default + 1M Reminder

### What changes

- Default context model is explicitly set to 200K Sonnet at init
- Two reminder trigger points surface the 1M context option when relevant
- `project.json` gains a `default_context` field

### project.json addition

```json
{
  "default_context": "200k"
}
```

Valid values: `"200k"`, `"1m"`. Set at `threadwork init` via new question:

> *"Which Claude context model will you use by default? (1) Sonnet 200K — standard, recommended (2) Sonnet 1M — extended context, higher cost"*

### Reminder Trigger 1: pre-tool-use.js

Before spawning any agent via `Task()`, check task complexity indicators:
- Target file count >= 6, OR
- Agent type is `tw-debugger` or `tw-planner`, OR
- Task description contains keywords: `refactor`, `architecture`, `migrate`, `redesign`

If `default_context` is `"200k"` and any trigger fires, prepend to the agent's injection block:

```
⚠️ CONTEXT ADVISORY: This task has high complexity indicators (6+ files / architectural).
If you encounter context limit issues, consider asking the user to switch to the 1M context model.
Current default: Sonnet 200K.
```

This is informational only — never blocks execution.

### Reminder Trigger 2: /tw:budget dashboard

When the token tracker estimates that any single agent in the current session consumed > 150K tokens, add an advisory line to the budget dashboard:

```
Context advisory: Agent tw-planner used ~165K tokens last session.
Approaching 200K context limit. Consider 1M model for complex planning phases.
```

### Files to modify

**`hooks/pre-tool-use.js`** — add complexity check before injection block composition.

**`lib/token-tracker.js`** — add `getHighContextAgents()` returning agents that exceeded 150K tokens.

**`hooks/session-start.js`** — read `default_context` from `project.json`, include in orientation block.

## Change 3: Dual Budget — Cost + Context, Estimate vs Actual

### What changes

The token budget system gains a parallel cost budget layer. Both are tracked, surfaced, and reported together. Session token budget default is recalibrated per model tier.

### Session budget recalibration

| Model | Context Window | Session Token Budget Default |
|---|---|---|
| Sonnet 200K | 200,000 | **400,000** (was 800K — corrected) |
| Sonnet 1M | 1,000,000 | **800,000** |
| Opus | 200,000 | **300,000** |

At `threadwork init`, if the user selects 200K model (the default), `project.json` is written with `"session_token_budget": 400000`. If 1M is selected, `800000`.

## Change 4: Model Switch Policy

### What changes

- New `lib/model-switcher.js` module
- `threadwork init` asks for switch policy preference
- `pre-tool-use.js` consults model-switcher before each agent spawn
- Switch log included in session handoff
- `/tw:model` and `/tw:model policy` commands

### New init question

> *"When Threadwork recommends switching model tiers (e.g., Haiku → Opus for complex debugging), how should it proceed?"*
> *(1) Auto — switch automatically, notify after the fact*
> *(2) Notify — propose switch with 10-second countdown, proceed unless interrupted [RECOMMENDED]*
> *(3) Approve — always ask for explicit confirmation before switching*

Skill-tier-influenced suggestion:
- Ninja → suggest `auto`
- Advanced → suggest `notify`
- Beginner → suggest `approve`

Stored in `project.json` as `"model_switch_policy": "notify"`.

### Task complexity → model tier mapping

| Complexity | Indicators | Recommended Model |
|---|---|---|
| Simple | 1–2 files, clear scope, no architecture | Haiku |
| Medium | 3–5 files, some logic | Sonnet |
| Complex | 6+ files, architectural, debugging, refactor | Opus |

## Change 5: Blueprint Delta Analysis

### What changes

- `lib/blueprint-diff.js` — new module
- Blueprint versioning stored in `.threadwork/state/`
- `/tw:blueprint-diff`, `/tw:blueprint-diff --since-phase <N>`, `/tw:blueprint-lock` commands

### Blueprint storage

At `threadwork init`, if the user provides a blueprint document path, it is stored as `.threadwork/state/blueprint-v1.md`. If no blueprint is provided at init, this file is absent and `/tw:blueprint-diff` will note "No baseline blueprint found — run `/tw:blueprint-lock` to establish one."

Each time `/tw:blueprint-lock` is run, the current blueprint content is snapshotted as `.threadwork/state/blueprint-v<N>.md` where N increments.

Blueprint version index: `.threadwork/state/blueprint-index.json`:
```json
{
  "versions": [
    { "version": 1, "date": "2025-02-01", "file": "blueprint-v1.md", "note": "Initial" },
    { "version": 2, "date": "2025-02-20", "file": "blueprint-v2.md", "note": "Added auth requirements" }
  ],
  "latest": 2
}
```

Blueprint files are committed to git — they are intent documents.

### lib/blueprint-diff.js

```javascript
/**
 * Load the most recent blueprint version from storage.
 * @returns {string|null} blueprint content or null if none stored
 */
export function loadLatestBlueprint()

/**
 * Snapshot current blueprint content as a new versioned file.
 * @param {string} content - blueprint markdown content
 * @param {string} [note] - optional note for the version
 * @returns {number} new version number
 */
export function lockBlueprint(content, note)

/**
 * Diff two blueprint documents and categorize all changes.
 * @param {string} oldContent
 * @param {string} newContent
 * @returns {Object} { additive: [], modifications: [], structural: [] }
 *
 * Change object shape:
 * {
 *   id: string,
 *   description: string,
 *   section: string,        // which section of the blueprint changed
 *   affected_components: string[],  // files/modules likely affected
 *   estimated_tokens: number,       // rough token cost to implement
 *   estimated_cost: number          // rough dollar cost
 * }

 * @param {Object} changes - output of diffBlueprints()

 * @returns {Object} { rework: [], free: [], deferred: [] }
 */

/**
 * Generate cost estimates for three migration options.
 * @param {Object} mappedChanges - output of mapChangesToPhases()
 * @param {Object} pricing - from token-tracker loadPricing()
 * @returns {Object} {
 *   restart: { tokens: number, cost: number, description: string },
 *   in_place: { tokens: number, cost: number, description: string, risk: string },
 *   phased: { tokens: number, cost: number, description: string, deferred_cost: number }
 * }
 */
export function estimateMigrationCosts(mappedChanges, pricing)

/**
 * Format the full diff analysis report as a readable string.
 * @param {Object} analysis - full analysis object
 * @param {number} [sincePhase] - if set, adds "Impact on remaining work only" header
 * @returns {string} formatted report
 */
export function formatDiffReport(analysis, sincePhase)

/**
 * List all stored blueprint versions.
 * @returns {Array} version objects from blueprint-index.json
 */
export function listBlueprintVersions()
```

### Change categorization logic

`diffBlueprints()` uses section-level diffing (not line-level):

- **ADDITIVE**: new sections, new agents, new commands, new fields that don't alter existing fields
- **MODIFICATIONS**: changed behavior descriptions, altered function signatures, modified data formats, updated thresholds
- **STRUCTURAL**: changed core execution model, altered hook event types, changed state file schemas, renamed core directories, changed agent spawn model

The function uses keyword heuristics and section heading analysis — it does not use an LLM. It is fast, deterministic, and runs in under 1 second.

### /tw:blueprint-diff output format

```
── Blueprint Delta Analysis ─────────────────────────────
Comparing: blueprint-v1.md → new-blueprint.md
Project state: Phase 2 of 4 complete. Phase 3 in progress.

ADDITIVE CHANGES (3) — can be added without touching existing code
  [A1] New /tw:analytics command (new command, no existing code affected)
       Affects: templates/commands/ — NOT STARTED phase work
       Est. cost: ~$0.15 to implement
  [A2] New tw-analyst agent (9th specialist)
       Affects: templates/agents/ — NOT STARTED phase work
       Est. cost: ~$0.20 to implement
  [A3] New spec domain: performance/
       Affects: .threadwork/specs/ — additive to existing spec library
       Est. cost: ~$0.05 to implement

MODIFICATIONS (2) — behavior changes to existing features
  [M1] Token budget threshold changed from 80% to 75% warning
       Affects: lib/token-tracker.js — DONE (Phase 1)
       Est. rework cost: ~$0.10
  [M2] Handoff Section 4 format change
       Affects: lib/handoff.js — DONE (Phase 1)
       Est. rework cost: ~$0.15

STRUCTURAL CHANGES (1) — core architecture shifts
  [S1] State file schema: project.json gains required 'team_mode' field
       Affects: lib/state.js, hooks/session-start.js — DONE (Phase 1)
       Est. rework cost: ~$0.40

──────────────────────────────────────────────────────────
MIGRATION OPTIONS

Option A — Full Restart
  Rebuild all 4 phases from scratch with new blueprint.
  Estimated cost: ~$12.00 | Clean slate | No technical debt
  Use when: structural changes are pervasive or > 40% of work is DONE

Option B — In-Place Migration
  Implement [M1], [M2], [S1] as rework on completed phases.
  Implement [A1], [A2], [A3] as new additions to upcoming phases.
  Estimated cost: ~$0.85 rework + $0.40 for additive = $1.25 total
  Risk: minor — handoff format change requires reviewing existing handoffs
  Use when: rework cost is < 20% of restart cost (it is 10% here)

Option C — Phased Adoption
  Implement additive changes [A1], [A2], [A3] now (~$0.40).
  Defer [M1], [M2], [S1] to a future blueprint version (v0.3.0 → v0.4.0).
  Immediate cost: ~$0.40 | Deferred cost: ~$0.65
  Use when: you want to keep momentum and can tolerate temporary inconsistency

Recommendation: Option B (in-place migration) — rework is minor and low-risk.

Type the option letter (A/B/C) to proceed, or press Enter to cancel.
──────────────────────────────────────────────────────────
```

After the user selects an option, `/tw:blueprint-diff` writes the decision to `.threadwork/state/blueprint-migration.json` and snapshots the new blueprint as the next version. It does NOT begin any code changes — the developer runs `/tw:plan-phase` or other commands to actually implement the accepted changes.

### /tw:blueprint-diff --since-phase <N>

Same analysis but with the header changed to "Impact on remaining work only (Phases N+)" and the `mapChangesToPhases()` call filtered to `sincePhase = N`. Changes that only affect completed phases are listed in a collapsed "Already complete — no action needed" section rather than removed entirely (for full visibility).

### /tw:blueprint-lock

```
Usage: /tw:blueprint-lock [note]
Example: /tw:blueprint-lock "Added analytics requirements"

Snapshots the current blueprint content as blueprint-v<N>.md.
Use before making multiple blueprint edits to establish a clean baseline for /tw:blueprint-diff.
```

Reads blueprint content from `REQUIREMENTS.md` + `ROADMAP.md` combined if no explicit blueprint file is provided, as a reasonable approximation of the current project intent.

## Migration Guide: v0.2.x → v0.3.0

### What `threadwork update --to v0.3.0` does

1. **Backs up** current `hooks/` to `.threadwork/backup/v0.2.x-hooks/`
2. **Appends** Threadwork block to `.gitignore` (idempotent)
3. **Creates** `~/.threadwork/pricing.json` if absent
4. **Updates** `hooks/pre-tool-use.js` — adds complexity check and model-switcher call
5. **Updates** `lib/token-tracker.js` — adds cost tracking functions and `model` parameter to `recordUsage()`
6. **Creates** `lib/model-switcher.js`
7. **Creates** `lib/blueprint-diff.js`
8. **Updates** `project.json` — adds `default_context`, `cost_budget`, `model_switch_policy` (with sensible defaults if not answered interactively)
9. **Creates** `.threadwork/workspace/sessions/` directory
10. **Updates** `THREADWORK.md` with new commands
11. **Recalibrates** `session_token_budget` if currently 800K and `default_context` is `"200k"` → changes to 400K with a prompt: *"Your session token budget was 800K (calibrated for 1M context model) but your default context is 200K. Recalibrating to 400K. Confirm? [Y/n]"*
12. **Prints** full summary of changes

### What is NOT changed

- All spec files
- All journals and handoffs
- All plan XML files
- Existing `token-log.json` entries
- Hook registrations in `settings.json`

### Manual steps after upgrade

1. Run `/tw:blueprint-lock` to establish your first blueprint baseline if you have a blueprint document
2. Review `~/.threadwork/pricing.json` and update model prices if Anthropic has changed them since this release
3. Review the new `model_switch_policy` setting — default is `notify`

## Key Design Decisions and Rationale

**Why recalibrate 800K → 400K for 200K model?**
800K was designed for the 1M context Sonnet model. A 200K context model session realistically handles 4–8 substantial agent invocations before hitting diminishing returns — each at 20K–80K tokens. 400K is a more honest default that still leaves 30% headroom for overhead. Developers who want more latitude can increase it at init.

**Why a global pricing.json instead of hardcoding?**
Anthropic updates model pricing periodically. Hardcoded prices would require a full `threadwork update` to fix; a user-editable file means developers can patch prices immediately without waiting for a package release.

**Why does /tw:blueprint-diff not start implementation?**
The analysis is a decision-support tool. Immediately modifying code after selecting an option would bypass the developer's chance to review the analysis, consult the team, or reconsider. The separation between "decide" and "implement" is intentional.

**Why is model-switch-log.json excluded from git?**
Switch logs are session-level operational data — not intent documents. The summary that matters (which tasks used which models) is captured in `token-summary.json` which IS committed. The raw switch log with timestamps and policy details is noise in git history.

**Why are blueprint files committed?**
Blueprint files are intent documents — they record what the project was supposed to be at a given point in time. Committing them means the project's design evolution is traceable through git history alongside the code it produced.
